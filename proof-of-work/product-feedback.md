## Methodology

This feedback was generated by autonomously exploring RevenueCat's API surface, SDK documentation, and developer resources from the perspective of an AI agent building and optimizing subscription apps. ## Feedback Item #001 — CRITICAL

**Title:** No Programmatic Offerings Management API

**Category:** API

**Problem:** Offerings and Packages — the core configuration for what users see and can purchase — can only be created and modified through the RevenueCat dashboard. The REST API v2 has no endpoints for CRUD operations on Offerings or Packages.

**Impact:** This is the single biggest blocker for agent-driven monetization. Without programmatic Offerings management, agents cannot:
- Create new pricing tiers based on market analysis
- A/B test offering configurations via API
- Automatically adjust packages based on real-time churn data
- Build self-optimizing subscription models
- React to competitor pricing changes programmatically

Every pricing experiment requires a human to log into the dashboard and manually configure the offering variant. This bottleneck negates the speed advantage of using agents.

**Proposed Solution:**
```
POST   /v2/projects/{project_id}/offerings
GET    /v2/projects/{project_id}/offerings
GET    /v2/projects/{project_id}/offerings/{offering_id}
PUT    /v2/projects/{project_id}/offerings/{offering_id}
DELETE /v2/projects/{project_id}/offerings/{offering_id}
POST   /v2/projects/{project_id}/offerings/{offering_id}/packages
PUT    /v2/projects/{project_id}/offerings/{offering_id}/packages/{package_id}
DELETE /v2/projects/{project_id}/offerings/{offering_id}/packages/{package_id}
```

**Evidence:**
- Multiple GitHub issues across RevenueCat repos requesting this
- Community forum threads about automation limitations
- Hit this limitation within 2 hours of API exploration

## Feedback Item #002 — HIGH

**Title:** Charts API Needs Real-Time Event Streaming

**Category:** API

**Problem:** The Charts/metrics API is pull-based with batch granularity. To detect trends, agents must poll at intervals, which is wasteful and introduces latency between signal and response.

**Impact:** Agent developers running continuous optimization loops need to know about churn spikes, conversion drops, or MRR milestones as they happen — not when they next poll. The current approach means:
- Wasted API calls from polling
- Delayed response to negative trends (churn spike at 2 AM goes unnoticed until next poll)
- Cannot build reactive systems (e.g., auto-trigger retention offer when churn exceeds threshold)

**Proposed Solution:** Two complementary approaches:
1. **Webhook events for metric thresholds:** Configure alerts like "notify when daily churn exceeds 5%" or "notify when MRR crosses $X." These fire as webhooks.
2. **Server-Sent Events (SSE) stream** for real-time subscription events (new purchase, cancellation, renewal, refund) — similar to Stripe's real-time events.

**Evidence:**
- Stripe's event streaming is frequently cited as a best practice
- Agent developers in forums consistently request real-time data access
- Current polling pattern generates unnecessary API load

## Feedback Item #003 — HIGH

**Title:** SDK Quick Start Examples Need to Be Self-Contained

**Category:** Documentation

**Problem:** SDK code examples in the docs are often excerpts from larger projects — they reference undefined variables, omit imports, or assume surrounding context that doesn't exist. For human developers, this is fine — they read the surrounding explanation and fill in the gaps. For agents, incomplete code is a failure point.

**Impact:** When an agent tries to use a RevenueCat code snippet:
- Missing imports cause compilation errors
- Undefined variables require guessing (agents hallucinate the wrong values)
- Context-dependent examples don't work when pasted into a fresh file
- The agent must read multiple doc pages and synthesize, which is error-prone

**Proposed Solution:** Create a parallel set of "agent-optimized" code snippets:
- Every snippet is fully self-contained (imports, initialization, complete function)
- Every snippet includes realistic placeholder values (not `<your_api_key>`)
- Every snippet can be saved to a file and executed without modification (given valid credentials)
- Structured as JSON/YAML blocks that agents can parse programmatically

Example format:
```json
{
  "snippet_id": "swift-check-entitlement",
  "language": "swift",
  "sdk_version": "5.x",
  "description": "Check if a user has an active premium entitlement",
  "imports": ["RevenueCat"],
  "code": "func checkPremium() async throws -> Bool {\n    let info = try await Purchases.shared.customerInfo()\n    return info.entitlements[\"premium\"]?.isActive == true\n}",
  "prerequisites": ["Purchases.configure() called in app launch"],
  "related": ["swift-purchase-package", "swift-restore-purchases"]
}
```

**Evidence:**
- Multiple instances of hallucinated imports and variables when using RC SDK examples
- Other developer platforms (Stripe, Supabase) are adopting structured snippet formats
- RevenueCat's existing MCP server initiative shows awareness of agent-facing needs

## Feedback Item #004 — MEDIUM

**Title:** Publish OpenAPI Specification for REST API v2

**Category:** API / Documentation

**Problem:** RevenueCat's REST API v2 documentation is presented as rendered HTML pages. There is no publicly available OpenAPI (Swagger) specification file that agents can parse programmatically.

**Impact:** Without a machine-readable API spec, agents must:
- Rely on HTML-scraped documentation (fragile, incomplete)
- Manually map endpoints, parameters, and response schemas
- Cannot auto-generate type-safe clients
- Cannot validate requests before sending them

**Proposed Solution:** Publish an OpenAPI 3.1 spec at a stable URL:
```
https://api.revenuecat.com/v2/openapi.json
```

Include:
- All endpoints with request/response schemas
- Authentication requirements
- Rate limit headers documentation
- Realistic example payloads for every endpoint
- Error response schemas with all possible error codes

**Evidence:**
- Industry standard — Stripe, Twilio, GitHub all publish OpenAPI specs
- Enables automatic SDK generation for any language
- The spec becomes documentation that agents can directly consume

## Feedback Item #005 — MEDIUM

**Title:** Experiment API for Programmatic A/B Test Management

**Category:** API

**Problem:** RevenueCat's Experiments feature (A/B testing for pricing/paywalls) is dashboard-only. Agents cannot create, monitor, or conclude experiments programmatically.

**Impact:** Combined with the Offerings limitation (#001), this means the entire experimentation workflow requires human dashboard interaction:
- Can't create an experiment variant via API
- Can't check experiment status or results via API
- Can't programmatically conclude an experiment and roll out the winner

For agents running continuous optimization, this is a significant friction point.

**Proposed Solution:**
```
POST   /v2/projects/{project_id}/experiments
GET    /v2/projects/{project_id}/experiments
GET    /v2/projects/{project_id}/experiments/{experiment_id}
PUT    /v2/projects/{project_id}/experiments/{experiment_id}
POST   /v2/projects/{project_id}/experiments/{experiment_id}/conclude
```

**Evidence:**
- Natural extension of Offerings API request (#001)
- Enables full programmatic optimization loop: analyze metrics -> create experiment -> measure results -> roll out winner -> repeat

## Feedback Item #006 — LOW

**Title:** Rate Limit Headers Should Include Retry-After

**Category:** API

**Problem:** When hitting rate limits, the API returns a 429 status but the `Retry-After` header behavior and rate limit bucket details aren't clearly documented. Agents need explicit guidance on:
- Requests per second/minute limits
- Which endpoints share rate limit buckets
- Whether limits are per-key or per-project
- The exact `Retry-After` value to respect

**Impact:** Without clear rate limit documentation, agents either:
- Over-throttle (wasteful, slow)
- Under-throttle and get rate limited (errors, retry storms)
- Implement custom back-off logic instead of respecting headers

**Proposed Solution:**
- Document rate limits explicitly in the API reference
- Always return `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset` headers
- Return `Retry-After` header on 429 responses
- Consider a higher rate limit tier for verified agent/automation use cases
